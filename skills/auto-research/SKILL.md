# Auto Research — Skill Optimization

An autonomous meta-skill that iteratively improves Claude Code skills through a baseline → mutate → evaluate → select → promote cycle. Mutations are generated by Opus via Max Proxy; grading uses Sonnet via Max Proxy. Winners overwrite the target skill's `SKILL.md` and are committed to `workflow-kit` following the git-safe protocol.

---

## Usage

```
/auto-research [skill-name] [--sweep] [--dry-run] [--help]
```

| Argument | Description |
|---|---|
| `<skill-name>` | Run optimization on a specific skill (e.g., `/auto-research cron`) |
| `--sweep` | Auto-select the best optimization target from the registry |
| `--dry-run` | Evaluate variants but do not promote or overwrite *(v2 — planned)* |
| `--help` | Print this usage reference |

**Examples:**
- `/auto-research cron` — optimize the `cron` skill
- `/auto-research --sweep` — let the system pick the highest-value target
- `/auto-research --help` — show usage

## How It Works

1. **Baseline** — Load the target skill's current `SKILL.md`. Run the binary eval suite against it. Record pass/fail counts and score.
2. **Mutate** — Opus generates N candidate variants of the instruction body. Each variant preserves the frontmatter `description` and structure but rewrites sections to improve clarity, scope, or triggering accuracy.
3. **Evaluate** — Sonnet grades each variant against the held-out eval split. Scores are recorded in the experiment DB.
4. **Select** — The variant with the highest score (above baseline + improvement threshold) is chosen as the winner. Ties go to the shorter variant.
5. **Promote** — The winning variant overwrites the target skill's `SKILL.md`. A git commit is created in `workflow-kit` with a structured commit message referencing the experiment ID.

If no variant beats baseline, no promotion occurs and the experiment is logged as `no_improvement`.

---

## Safety Gates

All gates must pass before a variant can be promoted:

- **Golden test** — A fixed set of "must trigger" and "must not trigger" examples. Any variant that fails a golden test is immediately disqualified.
- **Holdout split** — Evals are split 80/20. Variants are scored on the held-out 20%. This prevents overfitting to the training eval set.
- **Suspicious improvement flag** — If a variant scores >25% above baseline, it is flagged for manual review rather than auto-promoted. Extreme scores likely indicate eval leakage or prompt injection.
- **Cost budget** — Each cycle is capped at $0.50 in LLM spend (configurable in `registry.json`). If the budget is exceeded mid-cycle, the run halts and logs a `budget_exceeded` result.
- **Cache miss circuit breaker** — If the Max Proxy cache hit rate drops below 50% during an eval run, the cycle pauses and logs a warning. High cache miss rates inflate cost and indicate eval set instability.

## Scripts

> **Note:** Scripts are placeholders — implementation tracked in Phase 1–3 of the Auto Research plan.

| Script | Description |
|---|---|
| `scripts/run_cycle.py` | Full optimization cycle for a single skill: baseline → mutate → evaluate → select → promote |
| `scripts/sweep.py` | Registry-driven sweep: selects highest-ROI target skill and runs `run_cycle.py` on it |
| `scripts/mutate.py` | Calls Opus to generate N candidate variants from a source SKILL.md |
| `scripts/evaluate.py` | Runs binary eval suite against a variant; returns score and per-case pass/fail |
| `scripts/promote.py` | Overwrites the target SKILL.md with the winner and commits to `workflow-kit` |
| `scripts/budget.py` | Tracks per-run LLM spend and enforces cost caps |
